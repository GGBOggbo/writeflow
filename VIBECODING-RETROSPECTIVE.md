# Writeflow Vibecoding 复盘 — 给下一次开发提效的经验

> 基于 12 天 / 114 次提交 / 31 份设计文档 / 33 份实施计划 / 1451 行 findings 的真实轨迹反推。
> 不是方法论,是从你自己的提交记录和踩坑文档里挖出来的。

## 经验 1:警惕"设计文档成瘾"——docs 占 40% 是危险信号

### 你做对了什么
每次开发一个功能,你都遵循 `spec(设计) → plan(计划) → impl(实现)` 三件套,而且 findings.md 记录了真实的根因(比如 ISP 劫持 SMTP、积分绕过漏洞)。这套纪律本身是对的,**比 95% 的 vibecoder 都强**。

### 但踩进了一个陷阱
31 份 spec + 33 份 plan,光排版一个领域就有 **10+ 份设计文档**:

```
6-13 semantic-formatting → 6-13 formatting-v2 → 6-13 html-compatibility
→ 6-14 md2wechat-skeleton → 6-14 editorial-paper → 6-14 format-reverse
→ 6-14 restraint → 6-15 advanced-modules → 6-15 full-modules
→ 6-15 reference-alignment → 6-16 prompt-led → 6-17 writeflow-human
```

**每个文档都很认真,但它们在描述"同一个问题的不同解法",而不是"不同问题"。** 设计文档在这里变成了**推迟决策的工具**——写一份新 spec 比删掉旧代码心理成本低,于是不断推倒重来。

### 下次的规则
1. **一个功能领域,同时只允许存在一份"当前 spec"**。写新 spec 之前,必须先在旧 spec 顶部写明"为什么作废"——如果说不清楚,说明不该写新的。
2. **设一个"重构预算"**:同一个领域,第 3 次重写时强制停下来,花 30 分钟问"是不是这个功能根本不该自己做 / 该直接抄一个开源方案"。
3. **区分"探索性文档"和"执行性文档"**。前期可以多写探索性的,但进入实现后,新文档必须回答"它在哪个 Task 上推进",否则不许开新文件。
4. **docs 提交占比超过 30% 时,强制问一句:我现在是在推进产品,还是在回避写代码?**

## 经验 2:不要在一个"无明确终点"的功能上反复打磨

### 排版系统为什么是黑洞
看你的提交时间线,排版系统的演进路径是:

```
AI自由生成HTML → 混合(AI语义块+代码渲染) → 三主题 → 加restraint
→ md2wechat围栏解析 → editorial主题 → 反向边界 → 克制
→ 完整模块支持 → 模块对齐 → prompt驱动 → writeflow自研
```

**每一步看起来都在"改进",但终点一直在移动**。最初是"能复制到微信",后来变成"视觉原创",再变成"阅读续读",再变成"摆脱 md2wechat 复制"。**目标函数一直在变,所以永远达不到。**

### 下次的规则
1. **开工前写下"Done 的可验证标准"**。比如排版的 Done 应该是:"3 篇真实文章复制到微信后台,手机上看排版不塌"。达不到就改实现,**达到了就不许再开新 spec**。
2. **区分"核心功能"和"打磨功能"**。核心功能(7 步写作流、AI 调用、积分)你做得很稳;排版是"锦上添花",不该占 12 天里的一半时间。
3. **用"会话数预算"卡住打磨**:一个功能超过 3 个工作日还没收敛,要么砍掉,要么声明"够用了"。
4. 真正的产品价值在"帮人写出更好的文章",排版只是最后一公里。**别让最后一公里吃掉前 90%。**

## 经验 3:你的 TDD + 状态机纪律是真正的资产,保留它

### 这部分值得肯定
- **状态机是纯 reducer**(`transitionWorkflow`),18 个事件类型,前置门控清晰 → 这让工作流逻辑可测、可回放,极其正确
- **积分用 reserve→consume/refund + operationId 幂等** → 重新生成不会双扣,失败会退款,这是生产级设计
- **每个 Task 都是 RED→GREEN**:`progress.md` 里反复出现"RED:测试失败 → GREEN:n files passed" → 这是真在跑 TDD,不是装样子
- **双 DB 抽象**(sqlite/pg)让本地开发零配置 → 体验很好

### 下次保持
- 状态机/reducer 模式继续用。任何多步流程都该这样写。
- operationId 幂等模式继续用。任何"扣资源"的操作都该这样。
- RED→GREEN 继续严格做。这是你 fix 提交只有 16% 的根本原因(很多 bug 在测试阶段就被挡住了)。

## 经验 4:fix 提交暴露了三类系统性问题

分析 18 个 fix 提交,问题集中在三类:

### 类型 A:第三方集成的不确定性(占比最高)
```
- ISP 劫持 SMTP(QQ/163/gmail 全挂)
- Resend 免费版只能发给自己
- better-auth account_not_linked(邮箱验证状态、accountLinking)
- Google/GitHub OAuth 配置
```
**下次规则**:任何依赖第三方(邮件/OAuth/支付/搜索 API)的功能,**开工前花 30 分钟写一个 "spike"**——只验证"这条链路能不能通",不写业务代码。你的 findings.md 里 SMTP 那段就是事后补的 spike 笔记,如果开工前做能省半天。

### 类型 B:类型契约漂移
```
- fix(test): align test fixtures with evolved type contracts(37 个类型错误)
- TopicOption/OutlineSection/MaterialSlot 字段演进后测试夹具没跟上
```
**根因**:你用 `satisfies` 和 `as const` 很重,类型演进时旧测试夹具会批量失效。
**下次规则**:类型改了,**当次提交里就把所有测试夹具改完**,不要留到下个会话。37 个类型错误堆积说明有几次"改完类型就收工了"。

### 类型 C:排版相关的反复微调
```
- fix(formatting): restraint pass — drop quote-marks, white cta...
- fix(format-draft): constrain cta to at most one at the end
- fix(wechat-format-panel): source theme swatch color from tokens
```
**这些 fix 本身没错,但它们不该是 fix,而该是初版就纳入的规则。** 反映的问题是:**排版规则没有一次性写全,而是"看到难看就改一下"**。
**下次规则**:做"视觉/排版"类功能时,**先把所有约束列成一张表**(禁用 X、最多 N 个、首屏不能 Y),一次性写进 prompt 和 renderer,**不要靠 fix 慢慢补**。

## 经验 5:数据持久化策略——localStorage 是个限时炸弹

### 你做了什么
工作流状态**只存 localStorage**(`ai-writing-mvp-workflow:<userKey>`),服务端只存积分。

### 真实代价
- 用户清浏览器 / 换设备 → **所有草稿丢失**
- storage 里写了大量 `normalizeWorkflowState` 逻辑(回退 currentStep、修复孤儿 ID、丢弃坏快照)→ 因为 localStorage 数据天然不可信,你被迫写了 200+ 行防御代码
- `findings.md` 里专门记了"invalid localStorage snapshots restored without validation"这个 bug

### 下次的决策框架
| 数据 | localStorage 够吗 | 原因 |
|---|---|---|
| 积分/余额 | ❌ 必须服务端 | 可被篡改、需跨设备 |
| 认证 | ❌ 必须 cookie+DB | 安全边界 |
| **草稿/工作流** | **看产品阶段** | MVP 可接受,**但有用户后必须迁服务端** |
| UI 临时状态(当前 tab) | ✅ 够 | 丢了无所谓 |

**下次规则**:MVP 第一天就问"这个数据丢了用户会不会骂娘"。会的话,就算第一版用 sqlite 存服务端,也别只放 localStorage。Writeflow 现在如果要上生产,这是**最大的一笔技术债**。

## 经验 6:你其实有一套"隐性工作流",把它显性化

从 findings/progress 反推,你每个功能实际走过的流程是:

```
1. 遇到问题/有想法
2. (有时)写 spec → 写 plan
3. RED:先写失败测试
4. GREEN:实现到测试通过
5. 跑 npm test / lint / build
6. commit
7. 把决策和踩坑记进 findings.md
```

### 这套流程的强项
- 步骤 3-6(测试驱动 + 验证)执行得非常稳,几乎每次都做
- 步骤 7(findings 记录)质量很高,是真正的知识资产

### 这套流程的弱项
- **步骤 2 经常跳过或重复**(排版做了 10 份 plan)
- **缺少"决策闸门"**:没有一个环节强制问"这个值不值得做 / 够不够好了"

### 下次的显性清单(建议贴在显眼处)
每次开新功能前,过这 5 个问题,任何一个答不上来就不许写代码:
1. **Done 长什么样?**(可验证的验收标准,不是"更好看")
2. **这是第几次做同类功能?**(第 3 次以上要警惕)
3. **最小可删版本是什么?**(能砍掉哪些"也想做"的)
4. **依赖了哪个第三方?它的免费/限制边界在哪?**
5. **数据丢了会怎样?**(决定存哪里)

## 经验 7:具体的"下次直接照抄"的工程模式

这些是你已经验证过有效的模式,直接复用:

### ✅ 值得照抄
- **NDJSON 流式**(而非 SSE):因为 POST body + 流式返回,比 EventSource 适配现有 JSON API
- **operationId 幂等计费**:reserve→consume/refund,任何按次计费的场景
- **pure reducer 状态机**:多步流程 + 前置门控
- **Mock provider 默认回退**:`AI_PROVIDER=mock` 开箱即用,真 provider 缺 key 才报错
- **findings.md 格式**:日期标题 + 决策 + 根因 + 关键文件 + 验证状态(你的原生格式就很好)
- **spec→plan 配对文档**:每份 spec 配一份 plan,plan 用 Task + checkbox
- **双 DB(sqlite 本地 / pg 生产)**:零配置启动

### ❌ 下次别这样
- 不要让"排版/视觉"这类主观功能吃掉超过总时间 20%
- 不要在同一个领域堆超过 2 份未作废的 spec
- 不要把核心业务数据(草稿)只放 localStorage
- 不要靠 fix 提交来补视觉规则(一次性列约束表)
- 不要在改类型后不更新测试夹具

## 总结:你的 vibecoding 效率画像

```
强项(继续保持):
  - TDD 纪律强,fix 率低
  - 文档化决策和踩坑的习惯好
  - 状态机/幂等/降级等工程模式成熟
  - 本地开发体验好(双DB/mock)

弱项(下次改进):
  - 单一功能(排版)反复重做,目标漂移
  - 用写文档回避收敛决策
  - 第三方集成缺少前置 spike
  - 核心数据(localStorage)持久化策略偏弱

最大的单一提效点:
  → 给每个功能设"Done 标准"和"时间预算"
  → 排版这类打磨功能控制在 20% 时间以内
```

这份复盘的价值不在"我下次会更好"这种空话,而在**把上面 5 个闸门问题和 7 条规则变成下次开发时的硬性检查项**。
