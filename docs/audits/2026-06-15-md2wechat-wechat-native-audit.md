# md2wechat `wechat-native` Markdown 与预览审计

## 审计范围

- 用户提供的 411 行 Markdown 完整能力样稿。
- `https://md2wechat.app/theme-gallery/wechat-native` 主题详情页约 17,000px 的完整预览。
- 用户补充提供的约 156KB、1006 行完整转换 HTML，可与 411 行 Markdown 逐模块映射。
- 430px 手机视口下的开场、信息图、图片对比、价格卡等关键模块。
- 当前项目的 Markdown 解析、AI 语义分类、模块组合、主题渲染和微信兼容层。

注意：主题详情页可见区域不便逐项核验全部模块，但补充 HTML 已证明完整能力样稿的所有高级模块均有真实渲染结果。网页顶部主题介绍卡、页面底部接口说明和同组比较仍不属于文章主题本体。

## 核心结论

`wechat-native` 的核心不是一套绿色 CSS，而是三层系统：

1. 基础 Markdown 负责连续正文。
2. 结构化模块负责封面、证据、步骤、对比、案例和收尾等阅读任务。
3. 主题只负责颜色、字体、圆角、边框、阴影和间距。

当前项目已经具备第 1 层和一部分第 2 层，但只有 8 种语义块。若继续只调颜色，无法得到页面中 metrics、FAQ、图片对比、价格、系列等效果。

## 视觉审计

### 做得好的地方

- 正文保持 16px、约 1.7 行高和稳定段间距，重点模块才使用卡片。
- 同一套深绿、浅绿和灰色贯穿标题、边框、标签和状态，不靠大量颜色制造层次。
- 模块都有明确阅读任务：封面负责入口，metrics 负责结论，steps 负责行动，FAQ 负责消除疑问，CTA 负责收尾。
- 桌面双列和三列模块在 430px 下会堆叠为单列，图片和文字仍可读。
- 模块字段固定，内容变化不会导致骨架随机变化。

### 风险与限制

- 主题详情页在 430px 视口下总内容宽度约 602px，存在横向溢出。主要问题来自画廊外层展示容器，因此它不是严格的微信真机模拟器。
- 网页响应式依赖不能直接复制到公众号富文本。微信更适合内联样式、稳定 table 或天然单列布局。
- 完整样稿和转换 HTML 覆盖全部 31 种模块，但真实文章不应在同一篇里全部使用。样稿自己也建议每篇只放 3 到 6 个重点模块。
- pricing、logos、people 等模块更像服务页或发布稿组件，不适合默认自动塞进普通公众号正文。

## 当前项目真实差距

### 已经具备

- 安全 GFM 渲染和 Markdown 编辑。
- `heading / paragraph / quote / pain / transition / list / comparison / cta` 八类语义块。
- `hero / part / quote-card / callout / bridge / list-panel / comparison-panel / cta` 固定骨架。
- 强调密度限制、相邻装饰块降级、CTA 最多一个。
- 微信导出清洗、字体继承、列表拍平和纯图片 flex 转 table。

### 明确缺失

- `:::` 指令解析器。
- 能携带多字段的 `ArticleBlock` 联合类型。
- 图片组合、FAQ、steps、metrics、series 等确定性渲染器。
- 高级模块的微信端稳定降级规则。
- GFM 提示框类型和正确脚注处理。

运行验证表明：`:::metrics` 当前会原样显示为普通文本；`[!NOTE]` 只是普通引用；脚注会被错误解析为链接。

## 推荐借鉴顺序

### 第一批：直接提升公众号长文

1. `part`：章节编号、标题、可选副标题。
2. `summary / verdict / quote`：统一成少量结论型模块。
3. `toc / checklist / steps`：由大纲和列表确定性生成。
4. `notice`：支持 note、tip、important、warning、caution。
5. `faq`：适合教程和转化文章。
6. `image-text / image-compare`：解决教程、案例和产品文章的图片表达。
7. `subscribe / cta`：继续保持全文最多一个。

### 第二批：特定文章类型

- `metrics`、`audience-fit`、`cases`、`toolbox`、`series`、`author-card`。
- 必须由真实数据或用户资料填充，AI 不得自行编造。

### 后置支持

- `pricing`、`logos`、`people`、`gallery`、`longimage`、`dialogue`、`image-annotate`。
- 这些模块同样可以完整复刻，但使用频率较低、数据真实性风险较高或需要额外微信降级规则，建议在核心模块之后实现，而不是删除。

## 推荐架构

不要直接把 31 种模块塞进现有 `FormattingBlockType`，也不要让 AI 输出任意 HTML。建议增加独立的结构化层：

```text
Markdown 正文
→ 确定性 Markdown 解析
→ AI 只做有限的语义选择
→ ArticleBlock[]（带字段的联合类型）
→ 模块确定性渲染
→ 主题 token
→ 微信兼容降级
→ 预览与复制
```

第一版 `ArticleBlock` 只需要 paragraph、heading、quote、notice、summary、list、checklist、steps、faq、image-text、image-compare、cta。先覆盖公众号最常见的阅读任务，再考虑商业服务页模块。

## 最终判断

值得借鉴的是：**固定字段、固定模块、稀疏使用、主题与结构分离、手机端有确定性降级。**

可以完整复刻的是：31 个模块的 Markdown 语法、字段解析、确定性 HTML 骨架和主题样式。补充 HTML 已提供足够的输入输出证据。

不应该照搬的是：让 AI 在每篇文章里自动使用全部 31 个模块、依赖网页媒体查询的横向布局，以及把商业展示组件默认塞进普通公众号文章。
