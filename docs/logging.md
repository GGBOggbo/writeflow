# Logging

The app uses Pino JSON logs for server-side request and workflow tracing.

## Environment

- `LOG_LEVEL`: `error` by default. Use `info` for lifecycle logs and `debug` for topic/search diagnostics.
- `LOG_FORMAT`: production and Vercel should use JSON. Local development can use pretty output unless set to `json`.
- `LOG_HASH_SECRET`: optional HMAC secret for `userIdHash`. Set this in production if user correlation is needed.

## Correlation IDs

- `requestId`: generated by the server for every API request.
- `workflowId`: generated in the browser and persisted with the draft workflow. It is sent as `X-Workflow-Id` and confirmed back by the server.
- `operationId`: existing billing/idempotency ID from the request body.

JSON and streaming AI responses include `X-Request-Id` and `X-Workflow-Id` headers, so a browser request can be matched to Vercel logs.

## Safety Rules

Logs must not include API keys, cookies, authorization headers, full prompts, model response previews, article HTML, article bodies, comments, or raw user IDs. Search diagnostics may include titles, sanitized keywords, counts, route choices, timing, and rejection reason totals.

- `api.request.succeeded`
- `api.request.failed`
- `api.stream.closed`
- `credits.reserve.started`

- `credits.refund.succeeded`

- `ai.request.started`

- `search.article_info.failed`

- `search.comments.skipped`
- `search.comments.failed`

## Vercel Queries

Useful filters:

- `workflowId:<id>` to see one full writing workflow.
- `requestId:<id>` to inspect one API request.
- `operationId:<id>` to inspect one billable operation.
- `event:search.route.selected` to see whether wxrank used history, mixed, or realtime fallback.
- `event:search.provider.completed endpoint:getso` to inspect realtime search cost points.
- `event:search.context.prepared intent:topics` to confirm the reference pool sent into topic generation.

## Local Debugging

For detailed topic routing locally:

```bash
LOG_LEVEL=debug LOG_FORMAT=json npm run dev
```

Then reproduce one workflow and search the logs by `workflowId`, `event`, or `route`.
