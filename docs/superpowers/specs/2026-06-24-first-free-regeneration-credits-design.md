# 首次免费与重新生成 0.05 积分设计

## 背景

当前 Writeflow 的积分系统把 `topics`、`brief`、`outline`、`draft`、`meta` 五个 AI 生成阶段都视为计费阶段。普通用户初始 5 积分，每次成功生成扣 1 积分；失败时退款；管理员不消耗积分。AI 排版 `format-draft` 和 AI 补充素材 `complete-materials` 走免费接口，不进入积分扣减链路。

这套规则对早期测试简单，但现在有两个问题：

- 用户第一次走完整工作流会很快消耗额度，不利于体验。
- 用户真正应该付费的动作不是“第一次把阶段跑通”，而是“对同一阶段反复重新生成”。

本次设计把产品积分从“每次生成 1 积分”改为“每个工作流、每个阶段首次成功生成免费；同一工作流同一阶段重新生成扣 0.05 积分”。

## 目标

- `topics`、`brief`、`outline`、`draft`、`meta` 五个阶段，每个阶段在同一个工作流内首次成功生成免费。
- 同一个用户、同一个 `workflowId`、同一个阶段，从第二次成功生成开始，每次扣 0.05 项目积分。
- 新建工作流后，每个阶段重新拥有一次免费首生成。
- 0 积分用户仍然可以执行某个阶段的首次免费生成。
- 重新生成需要至少 0.05 积分；余额不足时阻止请求。
- 失败请求退回本次预扣金额；首次免费失败退回 0，重新生成失败退回 0.05。
- 系统内部自动重试不额外扣费；用户点击“重试失败请求”复用原 `operationId`，不重复扣费。
- AI 排版和 AI 补充素材继续免费。
- 服务端是计费事实来源，前端只做提示和体验优化。

## 非目标

- 不接充值、支付、订单、会员套餐或月度额度。
- 不把项目积分换算成人民币。
- 不新增 AI Token 成本统计。
- 不改变 DeepSeek 模型选择、wxrank 调用策略或搜索成本策略。
- 不把 AI 排版、AI 补充素材纳入计费阶段。

## 计费口径

计费维度是：

```text
userId + workflowId + stage
```

规则如下：

| 场景 | 扣项目积分 |
| --- | ---: |
| 同一工作流某阶段第一次成功生成 | 0 |
| 同一工作流同一阶段第二次及以后成功生成 | 0.05 |
| 失败请求 | 退回本次预扣金额 |
| 系统内部自动重试 | 不新增扣费 |
| 用户重试同一个失败请求 | 沿用原 `operationId`，不重复扣费 |
| AI 排版 / AI 补充素材 | 0 |

“首次”只看已经成功确认消耗的操作。失败过但没有成功产出的阶段，下一次仍然按首次免费处理。

## 数据模型

### 固定整数单位

数据库继续使用整数存储，但语义改为固定点单位：

```text
1 项目积分 = 100 credit units
0.05 项目积分 = 5 credit units
初始 5 项目积分 = 500 credit units
```

这样可以避免 SQLite、Postgres 和 JavaScript 浮点数之间的小数精度问题。

### 账户余额

`workflow_credit_accounts.balance` 存储 credit units。API 和界面展示时再转成项目积分小数，例如：

- 数据库存 `500`，界面显示 `5`
- 数据库存 `495`，界面显示 `4.95`
- 数据库存 `5`，界面显示 `0.05`

### 操作成本

`workflow_credit_operations.cost` 也存储 credit units。已有字段名可以保留，但实现时必须统一语义：`cost` 不再是“项目积分整数”，而是“credit units”。

### 工作流归属

`workflow_credit_operations` 需要记录 `workflowId`。现有客户端已经通过 `X-Workflow-Id` 请求头传递工作流 ID，日志上下文也会确认并返回它；积分系统目前还没有把它写入操作表。本次需要让 API 计费包装器把服务端确认后的 `workflowId` 传给积分存储。

历史操作没有 `workflowId` 时，应当视为旧规则遗留记录，不参与新规则下的“同一工作流同一阶段是否首次”判断，避免用户打开现有工作流时被旧数据误伤。

### 迁移安全

余额和成本从“项目积分整数”迁移到“credit units”时存在重复乘以 100 的风险。迁移必须有明确的一次性标记，例如 schema meta 表或等价的迁移版本记录：

- 未迁移时：旧余额 `5` 迁移为 `500`。
- 已迁移时：再次运行迁移不能把 `500` 变成 `50000`。
- 旧操作成本 `1` 迁移为 `100`，方便保留历史流水含义。
- 新增 `workflowId` 字段允许为空；新规则只统计非空且匹配当前工作流的成功操作。

## 预扣与确认流程

`creditStore.reserve()` 的输入需要包含：

- `userId`
- `stage`
- `operationId`
- `workflowId`

服务端流程：

1. 管理员直接返回无限额度，不扣费。
2. 如果 `operationId` 已存在，按已有操作返回幂等结果，不重新计算成本。
3. 如果是新操作，查询同一 `userId + workflowId + stage` 下是否已有成功确认的操作。
4. 没有成功确认操作：本次 `costUnits = 0`。
5. 已有成功确认操作：本次 `costUnits = 5`。
6. `costUnits = 0` 时允许 0 余额用户继续。
7. `costUnits > 0` 时检查余额是否足够，不足则拒绝。
8. 写入操作记录，并在需要时预扣余额。
9. AI 成功后 `consume()` 标记为成功确认。
10. AI 失败或流式中断后 `refund()` 只退回该操作记录里的实际 `costUnits`，并保证只退款一次。

这套流程保证“首次免费”由服务端根据成功操作判断，而不是由前端按钮状态猜测。

## API 与客户端体验

### API

五个计费路由继续使用现有的统一包装器，但包装器需要把确认后的 `workflowId` 传入积分层。JSON 和 stream 响应里的 `credits` payload 返回小数积分余额。

免费路由保持现状：

- `/api/ai/format-draft`
- `/api/ai/format-draft/stream`
- `/api/ai/complete-materials`
- `/api/ai/complete-materials/stream`

这些路由不得调用 `reserve()`、`consume()` 或 `refund()`。

### 前端

前端不能再用 `remaining === 0` 全局禁用五个生成按钮，因为 0 余额用户仍可能拥有该阶段首次免费生成资格。

前端应调整为：

- 按钮文案从“每次生成消耗 1 积分”改为“每阶段首次免费，重新生成扣 0.05 积分”。
- 初次生成按钮不因 0 余额直接禁用。
- 重新生成时可以根据本地状态做弱提示，但最终以服务端拒绝为准。
- 服务端返回积分不足时，提示“重新生成需要 0.05 积分”。
- 顶部积分卡、订阅页和说明页显示小数积分。

如果前端无法可靠判断某次点击是首次还是重新生成，也不要在客户端强行计算；让服务端计费结果决定。

## 文档与文案更新

实现阶段需要同步更新：

- `README.md` 中“每次生成消耗 1 积分”的描述。
- `docs/api-cost-ledger.md` 中产品积分口径。
- Landing page 中积分 FAQ 和功能卡片。
- 订阅页/账户页中“重新生成再次扣 1 积分”的说明。
- 各阶段按钮或空状态里的“积分不足”“消耗 1 积分”文案。

文档应明确区分：

- 项目积分：用户产品额度。
- wxrank 成本：公众号接口真实人民币成本。
- AI Token 成本：模型服务商按 token 收费，目前只给公式，不做实际流水统计。

## 测试策略

### 单元测试

- credit units 转换：初始 5 积分显示为 `5`，扣 0.05 后显示为 `4.95`。
- 同一 `workflowId + stage` 首次成功生成 cost 为 0。
- 同一 `workflowId + stage` 第二次成功生成 cost 为 5 units。
- 同一工作流不同 stage 仍然首次免费。
- 同一用户新 `workflowId` 下同一 stage 仍然首次免费。
- 0 余额用户可以首次生成。
- 余额低于 0.05 时，重新生成被拒绝。
- 失败请求按记录成本退款，0 成本和 5 units 成本都覆盖。
- 同一个 `operationId` 重试保持幂等，不重复扣费，也不把 0 成本重新计算成 5 units。
- 管理员仍然无限额度。

### API 测试

- 五个计费阶段的 stream / JSON 路由把 `workflowId` 传入积分层。
- 首次生成返回 credits payload，余额不减少。
- 重新生成余额减少 0.05。
- 积分不足时返回明确错误。
- `format-draft` 和 `complete-materials` 继续不产生 credits payload，不触发积分存储。

### 迁移测试

- 旧账户余额 `5` 被迁移为 `500` units。
- 旧操作成本 `1` 被迁移为 `100` units。
- 重复运行迁移不会二次乘以 100。
- 旧的 `workflowId = NULL` 操作不会让新规则误判为“已经生成过”。

## 风险与处理

- **迁移重复执行风险**：必须有一次性迁移标记，不能只靠字段类型判断。
- **并发双击风险**：同一用户同一工作流同一阶段并发生成时，可能同时判断为首次。实现时需要在事务内完成查询、插入和余额更新；必要时用数据库约束或锁保证同一计费判断原子化。
- **前端状态不可靠**：刷新页面、历史工作流、失败重试都可能让前端不知道是否首次。计费判断必须在服务端完成。
- **历史文案分散**：项目里已有多处“消耗 1 积分”文案，实现时需要全局搜索替换并结合上下文调整，不做机械替换。
- **小数展示混乱**：API 可以返回 number，但显示层需要统一格式，避免出现 `4.9500000001` 或 `4.9`/`4.90` 混用。

## 验收标准

- 普通用户余额为 0 时仍能在新工作流中完成每个阶段的首次生成。
- 同一工作流同一阶段第二次生成成功后，余额减少 0.05。
- 失败请求不会实际扣除积分。
- 系统内部重试和用户重试同一失败请求不会重复扣费。
- AI 排版和 AI 补充素材仍免费。
- 数据库余额和操作成本以整数 units 存储，界面以项目积分显示。
- 相关文档和产品文案不再声称“每次生成消耗 1 积分”。
