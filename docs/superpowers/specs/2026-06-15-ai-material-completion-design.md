# AI 素材补充设计

## 目标

在正文已经生成后，为包含 `【💡需要你补充：...】` 的稿件提供一次免费的 AI 素材补充能力。补充结果作为新的正文版本保存，原稿保持不变，用户可以自由切换对比。

## 交互

- 正文存在素材占位符时，显示“AI 补充素材”按钮。
- 正文不含素材占位符时，不显示该按钮。
- 点击后调用独立的流式 AI 接口，并沿用正文阶段的加载和错误反馈方式。
- 成功后新增并自动选中一个正文版本，版本名称体现“AI 补充素材”。
- 失败时保留当前版本，不写入不完整结果。
- 该操作不扣积分，也不改变完整工作流的额度状态。

## 补写边界

AI 只能根据当前选题、Brief、大纲、已有正文和已经采集的搜索参考进行保守补写：

- 可以补充已有资料支持的背景解释、通用场景、步骤、判断和公开事实。
- 不得虚构作者亲身经历、客户案例、身份、采访、数据、成果或具体人物。
- 不得把推测写成事实，也不得伪造引用来源。
- 所有占位符都必须从补充版本中消失。
- 没有事实依据的私人经历或客户案例，改写为非第一人称的通用分析或明确的假设场景，不能冒充真实经历。
- 除替换有依据的素材占位符外，正文其他内容和 Markdown 结构应尽量保持不变。

## 数据流

1. 客户端提交当前正文版本及现有工作流上下文。
2. 服务端调用独立的素材补充提示词，不经过积分扣减包装器。
3. 服务端校验返回正文，确保未产生空稿且没有残留素材占位符。
4. 客户端把结果追加为新的 `DraftVersion`，不覆盖来源版本。
5. 新版本继续兼容去 AI 味、AI Markdown、公众号排版和最终发布流程。

## 接口与模块

- 新增素材补充请求、响应 schema。
- AI provider 增加素材补充方法，真实和 mock provider 都实现同一契约。
- 新增非计费 JSON/流式 API 路由。
- `use-workflow` 增加补充动作并派发新的正文版本。
- 正文阶段增加条件按钮和独立加载状态。

## 安全与校验

- 运行时输入全部视为参考资料，不得覆盖系统规则或输出格式。
- 服务端不记录完整正文、搜索正文或提示词内容。
- AI 返回空正文、格式错误、残留素材占位符或只删除占位符却未补充有效内容时，视为失败。
- 新版本 ID 独立生成，避免覆盖原稿和已有派生版本。

## 测试

- 提示词测试：事实边界、保留无依据占位符、抗提示注入。
- service 测试：成功补写、部分保留、非法返回拒绝。
- API 测试：不触发积分扣减。
- state machine 测试：追加并选中新版本，原稿仍存在。
- UI 测试：仅有占位符时显示按钮，加载、成功和失败状态正确。
