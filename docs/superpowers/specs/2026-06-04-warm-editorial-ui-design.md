# Warm Editorial UI Design

日期：2026-06-04

## 背景

当前产品的功能链路已经完整，但界面的语气、层级和视觉组织仍更像“开发验证台”而不是“主编陪跑型 AI 共创工作台”。现有双栏结构可用，但左栏缺少真实控制职责，右栏缺少“编辑判断”质感，顶部区域也没有把单篇文章的推进感表达出来。

## 设计目标

1. 将首页体验从“原型面板”升级为“主编陪跑型工作台”。
2. 保留现有状态机与阶段流程，不改变真实产品心智。
3. 强化每一步的判断感、推进感和成稿感。
4. 在不引入复杂新依赖的前提下，建立一套更成熟的视觉系统与轻量动效节奏。
5. 把改动集中在 shell、阶段容器、全局样式和少量复用组件里，便于后续回退。

## 设计结论

采用 `Warm Editorial` 方向：保留暖色纸张感与中文长文产品的亲和度，同时引入更强的任务封面、主编台、结果工作台和阶段轨道，形成“编辑部桌面”体验。

## 信息架构

### 1. 顶部改为任务封面

顶部不再强调 MVP 状态，而是展示当前稿件的工作信息：

- 当前阶段
- 当前主题 / 想法摘要
- 产品主承诺
- 联网策略摘要
- 主动作按钮（开启新稿）

它应像一张有温度的“稿件工作卡”，既承担欢迎区，也承担流程概览。

### 2. 左侧改为主编台

现有 `ChatPanel` 重构为主编台，固定承担四类信息：

- 当前任务：这一步到底要产出什么
- 判断标准：用户应该看什么来做决定
- 策略信号：联网增强、质量警告、断网原则、结构类型等
- 下一动作：一个主按钮 + 少量次要提示

左侧不只是“说明”，而是“控制与判断辅助”。

### 3. 右侧改为成稿工作台

`ManuscriptPanel` 继续承载每个阶段的核心内容，但每个 stage 要有更明确的“审阅模式”：

- `Topic`：选题会
- `Brief`：写作策略单
- `Outline`：结构审稿
- `Draft`：版本对读
- `Meta`：包装定案
- `Final`：出稿面板

## 关键交互

### WorkflowStatus

从单纯的 pills 升级为“阶段轨道 + 节点语义”：

- 当前步骤：明确高亮，强调“你在这里”
- 已完成步骤：保留可点击回退能力，并显示已完成语义
- 未开始步骤：弱化显示，但维持整体流程可预见性

节点文案用短标签表达阶段状态，而不是只显示步骤名。

### Topic 阶段

卡片需要更像“选题会”的编辑卡：

- 主标题：切口名
- 子信息：切入角度
- 判断维度：核心观点、适合谁看、为什么值得写
- 信号信息：SEO / 质量警告 / 拥挤风险（若有）

被选中的卡片需要有明显“已入围”的状态表达。

### Outline 阶段

大纲与素材槽位需要变成“结构审稿视图”：

- 每段像一个章节模块
- 强调段落职责、核心推进与支撑素材
- 素材槽位像审稿边注，而不是平铺数据

### Meta 阶段

标题、摘要、封面建议应被组织成“包装决策”场景：

- 并列显示不同方案
- 明确当前选中状态
- 标签语气更像编辑判断而不是数据列表

## 视觉系统

### 色彩

- 继续使用暖中性色为基础
- 增加更深的墨色文本与更明确的金棕/橄榄辅助色
- 背景分层为：桌面底、纸张面、重点卡片面

### 字体

- 保留现有可用的 Next 字体加载方案
- 通过字号、字重、字距、大小标题节奏来制造“编辑感”
- 避免引入额外字体依赖，降低改动风险

### 卡片层级

定义三层卡片：

- 封面卡：首页顶部任务卡
- 工作卡：左侧主编台与右侧模块容器
- 结果卡：选题、章节、标题、摘要等内容卡

### 动效

只做低风险、轻节奏动效：

- 页面首屏淡入与上浮
- 阶段节点状态切换
- 卡片 hover / selected 强化
- 关键区块渐变和轻微光感

不引入重型动画库到业务代码中。

## 工程边界

### 允许改动

- `app/globals.css`
- `app/layout.tsx`
- `components/workspace-shell.tsx`
- `components/chat-panel.tsx`
- `components/manuscript-panel.tsx`
- `components/workflow-status.tsx`
- `components/search-toggle.tsx`
- `components/stages/*`

### 尽量不改动

- workflow state machine
- AI 调用链
- API routes
- 搜索 service

## HyperFrames 使用方式

本轮不把 HyperFrames 直接嵌入产品运行时，而是将它作为视觉演示与后续品牌化资产工具：

- 生成一份短场景 HTML 预览，用来表达 Warm Editorial 的视觉气质
- 为未来营销演示或方案展示提供同风格动态稿

这能满足“用上 HyperFrames 能力”的目标，同时避免把视频工具链硬塞进产品界面实现。

## 验收标准

1. 首页打开后，用户第一感知应更像“正在打磨一篇稿件”，而不是“在试一个原型”。
2. 左侧区域应清楚告诉用户当前步骤的目标、判断标准和下一动作。
3. 右侧三个关键高价值阶段 `Topic / Outline / Meta` 应明显比现状更具判断感。
4. 保持现有测试、lint 和 build 能通过。
5. 改动范围集中，便于人工对比与回退。
