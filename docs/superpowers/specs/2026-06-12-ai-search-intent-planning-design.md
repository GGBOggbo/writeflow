# AI 搜索意图规划设计

## 背景

当前选题搜索先通过规则从用户输入中提取关键词，再交给 wxrank 的历史库和实时搜索。该方式对简单中文主题有效，但会破坏产品型号、版本号和中英文混合实体。例如 `GPT-5.6` 会被压缩成 `gpt 痛点`，历史库进一步只搜索 `gpt`，从而召回军事、手机营销等低相关内容。

用户还可能输入 10 至 20 句话。此时直接搜索全文、截取固定长度或依赖正则提词，都无法稳定识别核心主题、对象和使用场景。

## 目标

- 每次生成选题前，使用当前 AI provider 解析用户的搜索意图。
- AI 解析包含在选题原有的 1 积分内，不额外计费。
- 保留用户原始创作意图，不允许搜索规划覆盖或改写选题主题。
- 历史库优先、实时搜索兜底的成本路由保持不变。
- 搜索规划或搜索服务失败时，降级为现有规则或无搜索参考生成，不阻塞选题。
- 明显跑题的搜索结果不得进入选题 prompt。

## 非目标

- 不使用 AI 对每一篇搜索结果逐篇重新评分。
- 不改变正文、策略单、大纲和标题摘要的积分规则。
- 本次只改选题阶段，不扩展 brief、outline 或 meta 的搜索意图规划。
- 不让 AI 自由决定调用哪个外部接口或调用次数。

## 总体流程

```text
用户原始输入
  -> AI 搜索意图解析
  -> 结构化 SearchPlan
  -> wxrank 当月历史库
  -> wxrank 上月历史库（合格结果不足时）
  -> wxrank 实时搜索（仍不足时）
  -> 相关性质量闸门
  -> 合格参考进入选题 prompt
  -> 生成 3 个选题
```

AI 只生成搜索计划。wxrank 路由、调用上限、去重和相关性过滤仍由确定性代码控制。

## SearchPlan 数据结构

```ts
type TopicSearchPlan = {
  coreTopic: string;
  historyKeyword: string;
  realtimeKeyword: string;
  requiredTerms: string[];
  relatedTerms: string[];
  excludedTerms: string[];
};
```

字段含义：

- `coreTopic`：对用户创作主题的单句压缩，仅用于日志、检查和搜索上下文说明。
- `historyKeyword`：适合 wxrank 历史库的较宽关键词，但必须保留核心实体。
- `realtimeKeyword`：适合实时接口的精确查询，保留版本号、产品名和关键场景。
- `requiredTerms`：准入词。搜索结果至少命中其中一个核心实体或可靠别名。
- `relatedTerms`：相关词。用于提高排序分，不单独构成准入条件。
- `excludedTerms`：排除词。结果主体命中这些明显错误方向时应被剔除。

示例：

```json
{
  "coreTopic": "GPT-5.6 对普通职场人的影响",
  "historyKeyword": "GPT 职场",
  "realtimeKeyword": "GPT-5.6 普通员工 职场转型",
  "requiredTerms": ["GPT-5.6", "GPT"],
  "relatedTerms": ["AI替代", "工作效率", "岗位转型"],
  "excludedTerms": ["军事", "国际政治", "手机广告"]
}
```

`excludedTerms` 不是固定黑名单，也不是看到某个词就无条件删除。它表示与本次用户意图明显冲突的主题方向。例如一篇文章同时提到 `GPT` 和 `军事`，只有当标题、摘要主体确实是军事内容时才排除，避免误伤正常类比或顺带提及。

## AI 解析策略

- 每次选题生成都调用一次，不按输入长度跳过。
- 复用当前配置的 AI provider，与选题生成使用相同供应商。
- 输出固定 JSON，并用 Zod 校验。
- 建议上限约 500 completion tokens，低温度，关闭思考输出。
- prompt 明确要求：
  - 保留产品型号、版本号、人名、公司名及中英文混合实体。
  - 长文本要识别主命题，不能把例子误判为主题。
  - `historyKeyword` 可以适度放宽，但不能只留下 `AI`、`GPT` 等过宽词，除非用户主题本身就是泛主题。
  - `excludedTerms` 只列高置信度的冲突方向，最多 5 个；不确定时返回空数组。
  - 不得补造用户没有表达的事实、身份或立场。

## 相关性质量闸门

对历史和实时结果统一执行：

1. 标题与摘要规范化，保留字母、数字、连字符和小数点。
2. 至少命中一个 `requiredTerms`，否则不得进入结果池。
3. 命中 `relatedTerms` 可提高排序，但不能替代核心实体准入。
4. 当标题和摘要主体明显命中 `excludedTerms`，且核心实体只被顺带提及时，剔除结果。
5. 历史结果合格数量不足 5 篇时，继续执行现有实时搜索兜底。
6. 实时结果过滤后仍为空，则返回 `empty/degraded`，选题不注入搜索上下文。

为避免 `requiredTerms: ["GPT-5.6", "GPT"]` 被宽词 `GPT` 放进大量跑题文章，过滤器需要区分精确实体和宽泛别名：精确实体优先；仅命中宽泛别名时，至少还需命中一个 `relatedTerms`，或标题主体与 `coreTopic` 的其他关键词一致。

## 降级与错误处理

- AI 搜索规划请求失败：使用现有规则生成查询计划。
- AI 返回非法 JSON：记录脱敏错误并使用规则计划。
- AI 返回空的核心词：使用原始输入构造安全计划。
- wxrank 历史库失败：保持现有 provider 错误处理。
- 实时搜索失败但已有合格历史结果：继续使用历史结果。
- 最终无合格结果：不向选题 prompt 注入低质量参考，直接进行基础生成。
- 所有日志不得输出 API key、OAuth secret 或数据库连接串。

## 积分与调用成本

- 用户点击一次“生成选题”，服务端仍只预扣 1 积分。
- 同一个计费操作内依次执行搜索意图解析、搜索、选题生成。
- 任意环节最终导致整次选题生成失败时，沿用现有退款机制。
- 搜索意图解析失败但成功降级并生成选题时，不退款，因为用户已得到完整选题结果。
- 不新增独立 operationId，不允许绕过或重复免费调用。

## 进度事件

在现有真实后端事件流中增加或细化：

- `理解创作意图`
- `理解创作意图完成`
- `扫描参考池`
- 后续事件保持现有命名

事件必须由真实步骤完成后发出，不使用前端模拟动画。

## 代码边界

- 新增独立搜索计划 schema、prompt 和 planner，不把逻辑继续塞进 `lib/search/service.ts`。
- AI provider 新增 `planTopicSearch` 能力，mock 与真实 provider 都实现。
- `generateTopics` 负责协调计费操作内的规划、搜索和选题生成。
- wxrank provider 接收结构化计划，而不是重新猜测用户意图。
- 现有规则提取保留为 fallback，并升级为保留版本号和混合实体的安全实现。

## 测试要求

- `GPT-5.6` 必须完整保留，不能退化为 `gpt`。
- `Claude 5`、`iPhone 18 Pro`、中文产品名和人名可正确规划。
- 10 至 20 句话能提炼主命题，例子不会取代主题。
- AI 解析失败时规则 fallback 能继续生成。
- 仅命中宽泛 `GPT` 且主题为军事的结果被过滤。
- 命中精确实体和相关场景的结果被保留并排在前面。
- `excludedTerms` 不会因为正文顺带提及而误杀相关结果。
- 历史结果不足时仍调用实时搜索；足够时不产生额外实时费用。
- 搜索结果全不合格时，不把跑题材料注入选题 prompt。
- 搜索意图解析不产生第二次积分扣费。

## 验收标准

- 输入 `GPT-5.6` 时，传给 wxrank 的实时词保留完整版本号。
- 参考池不再出现仅因宽泛 `GPT` 命中的军事或广告文章。
- 长文本输入能够生成可解释的历史词、实时词和准入词。
- AI 规划服务不可用时，用户仍能完成一次选题生成。
- 一次选题生成仍只消耗 1 积分。
